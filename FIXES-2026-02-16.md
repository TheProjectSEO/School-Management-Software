## Lessons Learned

1. **Always verify database schema** before writing queries
   - Use `information_schema.columns` to check column existence
   - Don't assume columns exist based on variable names in code

2. **PostgREST limitations with OR queries**
   - Can't use `or()` with FK-joined fields
   - Solution: Filter in JavaScript after fetching

3. **FK join syntax matters**
   - Use: `table!foreign_key_constraint_name(...)`
   - Check constraint name in database: `\d+ students` in psql

4. **Silent failures are dangerous**
   - Query errors were logged but not surfaced to user
   - Added debug logging to catch issues earlier

5. **Migration safety**
   - Always use `IF NOT EXISTS` checks
   - Make migrations idempotent (safe to run multiple times)
   - Include verification queries at end

## Fix #11: "No valid students selected" Error on Change Grade
**Issue:** After selecting a student and clicking "Change Grade," error popup shows "No valid students selected"

**Root Cause:**
DataTable component had a bug when using `rowKey` prop:
- When `rowKey` is set (e.g., `rowKey="id"`), row IDs are the actual data IDs (UUIDs like "abc-123-def")
- But `onRowSelectionChange` was treating keys as numeric array indices
- Called `data[parseInt(key)]` which fails for UUIDs:
  - `parseInt("abc-123-def")` returns `NaN`
  - `data[NaN]` returns `undefined`
- This caused `selectedStudents` array to contain `undefined` values
- Validation filter removed all undefined values → empty array → "No valid students selected"

**Affected Files:**
- `components/admin/ui/DataTable.tsx` (lines 88-99)
- `app/(dashboard)/admin/users/students/page.tsx` (handleBulkGradeChange validation)

**Solution:**

### A. Fixed DataTable Selection Logic
```typescript
// Before (BROKEN):
const selectedRows = Object.keys(newSelection)
  .filter((key) => newSelection[key])
  .map((key) => data[parseInt(key)]);  // Fails when key is UUID!

// After (FIXED):
const selectedRows = Object.keys(newSelection)
  .filter((key) => newSelection[key])
  .map((key) => {
    // If rowKey is set, key is the rowKey value (e.g., student ID)
    // Otherwise, key is the numeric index
    if (rowKey) {
      return data.find((row) => String(row[rowKey]) === key);
    }
    return data[parseInt(key)];
  })
  .filter((row) => row !== undefined);
```

### B. Improved Validation in handleBulkGradeChange
```typescript
// Added debug logging
console.log("selectedStudents:", selectedStudents);
console.log("selectedStudents.length:", selectedStudents.length);

// Early check before setting loading state
if (selectedStudents.length === 0) {
  alert("No valid students selected");
  return;
}

// Simplified filter (no longer needed with DataTable fix, but safer)
const studentIds = selectedStudents
  .filter((s) => s && s.id)  // Simpler than complex type guard
  .map((s) => s.id);
```

**How DataTable rowKey Works:**
1. When `rowKey="id"` is set, TanStack Table uses `getRowId: (row) => String(row.id)`
2. This means row IDs in `rowSelection` state are student IDs (UUIDs), not indices
3. Example rowSelection state:
   ```javascript
   {
     "a1b2c3d4-uuid": true,
     "e5f6g7h8-uuid": true
   }
   ```
4. Must use `.find()` to match rowKey values, not array indexing

**Result:**
- ✅ Student selection now works correctly
- ✅ "Change Grade" successfully updates selected students
- ✅ "Move to Section" also benefits from fix
- ✅ "Deactivate" bulk actions work properly
- ✅ All DataTable instances using rowKey are now fixed

**Pattern for Future Reference:**
When using TanStack Table with custom `getRowId`:
- Row selection keys are the custom IDs, NOT array indices
- Must use `.find()` to locate data by matching the rowKey
- Don't use `parseInt(key)` on non-numeric IDs

## Fix #12: Deactivate Students Not Working - Status Not Visible in UI
**Issue:** After deactivating students, they still appear as "Active" in the list

**Root Cause:**
The `bulkUpdateStudentStatus` function only updated the `students` table, but not the `school_profiles` table. The list view queries `school_profiles.status`, so changes weren't visible.

**Why Two Tables?**
- `students.status` = Enrollment status (active, inactive, graduated, transferred)
- `school_profiles.status` = Account status (displayed in UI)
- Both need to be updated together for consistency

**Affected Files:**
- `lib/dal/users.ts` (bulkUpdateStudentStatus function)
- `app/api/admin/users/students/bulk-status/route.ts` (missing "suspended" status)

**Solution:**

### A. Updated bulkUpdateStudentStatus to Update Both Tables
```typescript
// Before (BROKEN):
const { error } = await supabase
  .from('students')
  .update({ status, updated_at: new Date().toISOString() })
  .eq('id', id);

// After (FIXED):
// 1. Get student's profile_id
const { data: student } = await supabase
  .from('students')
  .select('profile_id')
  .eq('id', id)
  .single();

// 2. Update students table
await supabase
  .from('students')
  .update({ status, updated_at: new Date().toISOString() })
  .eq('id', id);

// 3. Update school_profiles table (this is what shows in UI)
await supabase
  .from('school_profiles')
  .update({ status, updated_at: new Date().toISOString() })
  .eq('id', student.profile_id);
```

### B. Added Missing "suspended" Status
```typescript
// Before:
const validStatuses = ["active", "inactive", "graduated", "transferred"];

// After:
const validStatuses = ["active", "inactive", "suspended", "graduated", "transferred"];

// Also updated TypeScript type:
status: 'active' | 'inactive' | 'suspended' | 'graduated' | 'transferred'
```

**How the Fix Works:**
1. **Fetch profile_id** - Query students table to get the linked profile_id
2. **Update students table** - Update enrollment status
3. **Update school_profiles table** - Update account status (visible in UI)
4. **Error handling** - If any step fails, continue to next student and log error

**Status Values:**
- `active` - Student is currently enrolled and attending
- `inactive` - Student account is deactivated
- `suspended` - Student is temporarily suspended
- `graduated` - Student has completed their studies
- `transferred` - Student transferred to another school

**Result:**
- ✅ Deactivating students now works correctly
- ✅ Status changes appear immediately in the list
- ✅ Both database tables stay in sync
- ✅ All 5 status values are now supported
- ✅ Proper error handling for each step

**Pattern for Future Reference:**
When updating student data that affects the UI display:
1. Always check which table the list view queries from
2. Update both `students` and `school_profiles` tables if needed
3. Join students → profile_id → school_profiles
4. Keep both tables synchronized

This same pattern applies to:
- Student name changes (school_profiles.full_name)
- Email changes (school_profiles.email)
- Phone changes (school_profiles.phone)
- Avatar changes (school_profiles.avatar_url)
